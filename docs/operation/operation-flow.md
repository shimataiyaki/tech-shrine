# tech-shrineプロジェクト 開発・運用フロー

本文書は、tech-shrineプロジェクトにおけるプロダクトの開発・運用、およびフィードバック対応に関する手順を定めたものです。

## 1. 用語の定義

| 用語 | 定義 |
| :--- | :--- |
| **開発者** | 本プロダクトの著作権者であり、個人探究「tech-shrineプロジェクト」の実施主体。Shimataiyaki。 |
| **プロダクト** | 開発者が個人で開発し、MITライセンスで公開するソフトウェア群「digital-ema」「binary-omikuji」。 |
| **プロジェクト** | 開発者がプロダクトを部活動に提供し、その運用を通じて公共空間における人とテクノロジーの関わりを検証する個人探究活動の総称。 |
| **部活動** | プロダクトを文化祭展示のために運用する部活動。 |
| **部長** | 部活動の代表者。展示運営に関する現場責任者であり、プロダクトに対する要望の唯一の窓口。 |
| **顧問** | 学校の教員。部活動の教育活動としての管理責任を負うが、プロダクトの内部仕様に対する指揮命令権は持たない。 |
| **来場者** | 文化祭に来場し、プロダクトを体験する個人。 |

## 2. フィードバック対応フロー

プロダクトに関する要望・助言は、すべて以下のフローで処理します。

graph TD
    A[フィードバック発生源] --> B{種類は？}
    B -->|致命的な不具合| C[開発者が即時対応・事後報告]
    B -->|展示内容に関する要望・助言| D[部長が窓口]
    D --> E{部長の判断}
    E -->|採用| F[開発者が実現可能性を検討]
    E -->|不採用| G[部長から提案元へ説明]
    F --> H{技術的・運用的に可能か}
    H -->|可能| I[開発者が対応]
    H -->|不可能| J[開発者から部長へ理由を説明]

### フロー運用の原則

1. **窓口一本化**：プロダクトに関するすべての要望・助言は、一度**部長**を経由する。開発者は部長以外からの直接指示では動かない。
2. **判断の簡素化**：部長の判断は「採用する／しない」の二択。技術的詳細は開発者が引き受ける。
3. **緊急時例外**：展示が物理的に停止する致命的な不具合は、開発者の判断で即時修正し、事後報告とする。
4. **フリーズ期間**：文化祭直前の機能追加・仕様変更は、安定運用を優先し、原則として受け付けない（利用規約第4条第3項に基づく）。
